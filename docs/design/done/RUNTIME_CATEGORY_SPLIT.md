# Runtime Phase 2 — Category Split for Large Modules

Status: design · 2026-04-22

## Intent

Follow up on the Phase 1 runtime test-extraction by splitting the
remaining large production files in `crates/runtime/src/` into per-
category sub-modules. `string_ops.rs` (750 L, 22 FFI functions) is the
most obvious candidate — one file currently mixes string length, case
conversion, slicing, search, JSON escape, parsing, and C-string FFI
helpers. Other candidates in descending size: `variant_ops.rs` (730),
`signal.rs` (647), `list_ops.rs` (645), `weave.rs` (638), `scheduler.rs`
(688), `arithmetic.rs` (594), `crypto.rs` (588), `closures.rs` (573).

This design covers `string_ops.rs` specifically; the same pattern
applies to the others when a natural category axis exists.

## Constraints

- **FFI symbol names are frozen.** Every `pub unsafe extern "C" fn
  patch_seq_X` is a linker contract called by name from generated LLVM
  IR. Module path doesn't matter (the `#[unsafe(no_mangle)]` attribute
  detaches the symbol), but the function *name* must stay identical.
- **No behavior change.** 60 existing `string_ops::tests` tests must
  continue to pass byte-for-byte.
- **No public API change.** Any Rust callers that import via
  `crate::string_ops::X` keep working. The aggregator `string_ops.rs`
  (or `string_ops/mod.rs`) re-exports from the sub-modules.
- **Tests stay in `string_ops/tests.rs`.** Phase 1 already moved them
  there; this split is purely production-side.
- **Internal helpers** like `global_string` imports stay accessible via
  `use crate::seqstring::global_string` in each sub-module — no
  visibility widening needed.
- Out of scope: renaming FFI functions, changing stack conventions,
  adding new ops, unwrap-hygiene cleanup (tracked separately).

## Approach

Convert `string_ops.rs` to `string_ops/mod.rs` (or keep `string_ops.rs`
as the sibling aggregator — both work in Rust 2024, pick whichever
matches the existing runtime layout). Split the 22 FFI functions into
roughly 5 category files:

- `basic.rs` — length, byte-length, empty, equal, concat
- `case.rs` — to-upper, to-lower, trim, chomp
- `access.rs` — char-at, substring, char-to-string, find, contains,
  starts-with, split, join
- `conversion.rs` — string-to-int, json-escape, symbol-equal
- `cstring.rs` — string-to-cstring, cstring-to-string (FFI glue for
  C-convention callers)

Each sub-module file:
- imports `crate::seqstring::global_string`,
  `crate::stack::{Stack, pop, pop_two, push}`, `crate::value::Value`
  as needed.
- holds its category's `pub unsafe extern "C" fn patch_seq_*` entries
  with `#[unsafe(no_mangle)]` attributes unchanged.

`string_ops/mod.rs` declares each sub-module and adds `pub use X::*;`
for each, so the aggregator's public surface is identical to today's
flat file.

The existing `string_ops/tests.rs` doesn't need any change — its
`use super::*;` continues to reach every public fn through the
re-exports.

## Domain Events

- **Split lands** → `string_ops/` tree grows to aggregator +
  5 category files + tests. Every production file under ~200 L.
  Adding a new string op is a single-file, two-line edit (plus one
  test).
- **Linker integrity** → `nm`-based diff on the compiled runtime
  static library shows identical exported symbol names before vs.
  after the split. The LLVM IR generated by `seqc build` contains
  identical `declare ptr @patch_seq_string_*` lines (the runtime-
  declarations split we already landed doesn't need to change).
- **Precedent set** → same technique unlocks follow-up splits for
  `variant_ops`, `list_ops`, `weave`, `scheduler`, `arithmetic`,
  `crypto`. Each gets its own PR using this same template.

## Checkpoints

1. `just ci` passes — fmt, clippy `-D warnings`, unit tests,
   integration tests, seq lint all green.
2. Runtime tests: the same 60 `string_ops` tests pass (via
   `cargo test -p seq-runtime --lib`). The top-level `just test`
   number for seq-runtime is unchanged.
3. FFI symbol diff: `nm target/release/libseq_runtime.a | sort` is
   byte-identical (or differs only in section ordering) before vs.
   after.
4. Compile + run one example (`hello-world.seq`) and confirm it
   produces identical output.
5. Every new category file is ≤200 L. The aggregator
   `string_ops/mod.rs` is ≤50 L.
